About BOFS

How BOFS came to be, who built it, and how it's maintained.

Lineage

BOFS development started in 2015 at the University of Saskatchewan's HCI lab. The lab was beginning to conduct online studies, and because we wanted to incorporate interactive tasks built in Unity and served over the internet via Unity's web player, we could not use any off-the-shelf solution. Instead, each study was developed as a standalone web application. Every new questionnaire had to be written by hand using typical web technology at the time (laying out the inputs in HTML, defining a database table, etc.). To create a new study, the source code from the old study was copied and used as a starting point; each project required its own complete copy of the underlying source code. Any improvements to the underlying study flow behavior, questionnaire logic, presentation, integrated tasks, and so on, were tricky to share between projects, and it was time-consuming to keep old projects up to date. We called any system built using this approach a "frankensystem".

To improve the situation, we developed a "sequel" to the frankensystem, the "bride of frankensystem". We established a JSON schema that we could use to define questionnaires, and a shared Python package that contained universal participant and survey-related logic and page templates. We used the SQLAlchemy ORM library to dynamically create the database schema based on loaded questionnaire JSON files. We switched to Flask, which has the concept of "blueprints" to bundle related pages and behavior, and leveraged that to create new task integrations.

Everyone in the lab therefore shared the same base implementation for their study, and defined the unique aspects of their study using configuration files, plain-text questionnaire files, and modular folders for any additional behavior or task integration that the study required. This approach meant that every study could benefit from changes to the shared foundation.

Transition to Open Source

After a few years of internal use (resulting in 21 peer-reviewed publications that leveraged BOFS), we open-sourced the project in late 2019, and I (Colby) have maintained it individually ever since. Since going open source, it has been used in at least 40 publications, by researchers at universities such as Eindhoven University of Technology, University of Twente, Utrecht University, University of Victoria, and University College London. I maintain it to this day because I want to contribute in my own way to research and I have an interest in developing tools related to automating tasks. It just so happens that BOFS provides interesting design and development challenges for me.

Sustainability

BOFS leverages popular and well-maintained dependencies (Flask, SQLAlchemy, Pandas, etc.). This means that even without active development from me, the underlying packages that BOFS uses continue to receive security patches and enhancements. A self-hosted study deployed today can continue to run indefinitely and receive free, ongoing updates provided that the underlying Python packages on the server are periodically updated. BOFS is also open source (LGPL-3.0 licensed), and uses no propriety formats. This means that the project can be forked and updated as-needed, and existing projects will almost certainly work with the forked version. Data is stored in industry-standard database systems (like SQLite) that have their own well-supported tools. Any changes I make to BOFS today are generally to provide additional functionality or quality-of-life improvements — the core logic is thoroughly tested across thousands of participants.

I do not seek to monetize BOFS directly. I pledge that the main project will always remain open-source, and I will never charge licensing fees nor require payment to use the open-source project. At the moment, I am investigating the feasibility of monetizing it indirectly, by providing additional external tooling to help make BOFS easier to use or deploy (which I'm implementing via the "Frankenmanager").

If you want to support the project financially, you can sponsor the project on GitHub. If you want to create an online study but need additional help, or want to implement something beyond the current capabilities of BOFS, you can reach out to me about potential consulting work. For minor improvements that would be of benefit to everyone using the project (or to file bug reports), open an issue on GitHub or ask about it on Discord.
