Bride of Frankensystem

BOFS is an open-source framework for building online behavioral experiments and studies that survey platforms can't accommodate. Present Unity WebGL games, jsPsych and lab.js tasks, and custom JavaScript alongside questionnaires seamlessly, all on the same platform. Let BOFS provide the supporting infrastructure — participant routing, condition assignment, and data storage — you focus on your research.

Who BOFS is For

Behavioural scientists, HCI labs, games-research groups, and anyone running online studies. Whether the study is a short questionnaire or a multi-session design with custom tasks, balanced condition assignment, and trial-by-trial data captured alongside responses, the same BOFS project structure handles it — so the study can grow without switching tools.

Where BOFS Fits

If you've used other tools for online studies, here's how BOFS compares to them.

vs. Survey platforms (Qualtrics, SurveyMonkey, Google Forms)

Survey platforms handle questionnaires through a point-and-click editor on their hosted infrastructure. Your work lives in their UI and you can't embed bespoke JavaScript tasks, capture detailed logs from those tasks, or take your questionnaires with you when the subscription ends.

vs. JavaScript libraries (jsPsych, lab.js)

jsPsych, and lab.js handle in-browser trial logic — precise timing, key capture, randomization. They don't host the surrounding study (consent, condition assignment, sessions, admin panel). A jsPsych, or lab.js task can run inside a BOFS custom page; the two are complementary.

vs. Building from scratch (Flask, Express)

Building from scratch gives you full control, but you build every detail of the study scaffolding yourself. BOFS supplies that scaffolding and stays out of your way when you need to go beyond it. Custom Flask routes, blueprints, and templates drop into the same project, with full access to the database, sessions, and configuration.

How it Works

A BOFS project is a folder of plain-text files — TOML, JSON, and HTML — that describe your study. Version-control them, share them with colleagues, reuse them across projects, or just edit them in any text editor. Studies built years ago still run today with only minor adjustments.

Define your study flow

One TOML file: page list, conditions with random assignment and balancing, external IDs for MTurk or Prolific, and multi-session designs where participants resume their condition on a follow-up visit.

Write questionnaires

JSON files with item shuffling, conditional questions and pages, computed scores, and reusing earlier answers in later wording. Tag a questionnaire to repeat it in the same session for pre/post designs.

Embed a custom task

Drop in HTML/JavaScript tasks: Unity WebGL, jsPsych, lab.js, PsychoJS, p5.js, or your own.

Capture trial-by-trial data

Define custom data tables in JSON; POST rows from your task and BOFS stores them alongside questionnaire responses.

Monitor and export

Live admin panel: per-participant progress, per-condition descriptive stats with box plots, CSV / per-table CSV / full SQLite export.

Extend with Python

When built-in patterns aren't enough, custom Flask routes drop into the same project — full access to the database, sessions, and templates.

Ready to try BOFS?

Install with pip and run your first study locally. The documentation walks through setup, configuration, and your first questionnaire.

    BOFS 2.1

    The next BOFS release, version 2.1, is available for testing now. Grab the development release from PyPI: pip install bride-of-frankensystem-dev.

    Changes

    Getting Started

    • New BOFS init CLI wizard for creating a new BOFS project; can launch the project with the browser opened automatically.
    • SECRET_KEY is no longer required in the config; one is auto-generated and stored when the database is initialized.
    • Major overhaul of the documentation, including expanded config options and util function references, blueprint guidance, and docs for the title attribute, required_selection, and the init command.

    New Question Types and Questionnaire Features

    • Added a "video" question type for embedding video stimuli.
    • Added a "picture_select" question type for choosing one option from a set of images. Supports horizontal (default) or vertical centered layouts, a per-question width, and an auto_resize flag that normalizes images to the smallest natural dimension in the group. The stored column's data type is auto-detected (integer/float/string) from the image values, and the images list is validated at startup.
    • Checklist now supports options with associated text fields.
    • Radiolist now supports an "other" option with a free-text field, and can require a specific option to be chosen to continue.
    • Radiogrid can now be made optional, supports questions as an alias for q_text, and gives clearer validation feedback.
    • Questions can define a title attribute that displays above the question box.
    • Questions can be configured to show only for specific conditions.
    • Questionnaire questions are now loaded server-side instead of via htmx.
    • Questionnaire validation issues are reported on the console, and the user is directed to a friendlier error page.
    • If a participant returns to a prior questionnaire (supported internally, not exposed in the UI), their previously-entered values are reloaded into the form.
    • New QuestionnaireInteraction event log captures focus/blur/change/paste/visibility events for every input on every questionnaire (replacing the radiogrid-only click log). Text inputs also record per-field authenticity signals — keystrokes, backspaces, pastes, pasted character count, final length, total focus duration, and time-to-first-keystroke.

    Conditions

    • Can optionally load conditions from a .csv file or an existing .db file, making longitudinal studies easier to set up.
    • Individual conditions can be "paused" to stop new participants from being assigned to them.
    • If all conditions are paused, new participants are blocked from entering the study.

    Admin Panel

    • New procedure diagram visualizing the page flow.
    • Can preview an individual participant's responses.
    • Admin panel redirects users back to the log-in page when their session is no longer valid.
    • Refined admin control panel shown at the bottom of pages when logged in as admin or running in debug mode.
    • Export button on the database table view moved to the top of the page.

    Debug Mode and Developer Experience*

    • In debug mode, a condition picker lets developers choose which condition to test.
    • In debug mode (or via localhost), users can restart the experiment from /end; the restart route keeps the admin login session intact.
    • The debug toolbar is now shown on pages before the participant record exists, but with reduced functionality.
    • A console warning is shown when @verify_correct_page is missing on a custom route.

    Custom Routes and Templates

    • Activity polling is now injected via after_request, so participants on long-running tasks with fully custom HTML are no longer falsely marked as abandoned. Pages are also marked "submitted" when the session's current URL is updated.
    • Participant progress tracking moved out of @verify_correct_page into a before_request_ handler, so custom routes that omit the decorator still track progress properly.
    • Template's header colour can be set from the config file.
    • Alternative style.css files can be loaded via configuration.
    • APPLICATION_ROOT can now be omitted from the config.

    Expression Engine and Show-If

    • Added an expression engine for participant_calculations, replacing the use of eval().
    • Added show_if predicates on questionnaire questions (live, browser-side, against same-page form values) and on entries in PAGE_LIST (server-side, against stored answers). See docs/advanced/expressions.rst for the syntax.
    • conditional_routing now accepts an optional show_if predicate alongside (or instead of) condition.
    • All user-facing strings in questionnaire JSON (top-level title/instructions, per-question title/instructions/text/placeholder/left/right/labels/q_text[].text, etc.) are scanned for {{ expression }} placeholders at render time. Each placeholder is evaluated as a BOFS expression against the current participant and substituted (HTML-escaped) into the rendered HTML.

    Participant Data Access in Templates and Custom Routes

    • participant is now available as a template variable in every BOFS template; resolves to None when no participant is in session.
    • New participant.evaluate(expression) runs a BOFS expression against the participant's stored data. The same syntax used in show_if and participant_calculations is now usable from templates and custom blueprint code.
    • New participant.has_questionnaire(name, tag='') distinguishes a real submission from the blank-default row that participant.questionnaire(name) returns when nothing has been submitted yet.
    • participant.table(name) now returns a TableAccessor exposing the participant's raw rows (accessor.rows), every export aggregate as an attribute, and all aggregates as a dict (accessor.exports). Scalar exports resolve to a single value; group_by exports resolve to a dict keyed by the group value (or by a tuple when group_by is a list of columns).

    Security and Hardening

    • Added IP-based brute-force protection on the admin login.
    • Admin sessions are bound to the IP they were issued for. Participant sessions are bound by default and can be opted out via SESSION_BIND_TO_IP_PARTICIPANT = false.
    • New BEHIND_REVERSE_PROXY config (default false). When true, BOFS reads the real client IP from X-Forwarded-For via Werkzeug's ProxyFix; otherwise X-Real-IP and X-Forwarded-For are ignored. Enable this when running behind Caddy or nginx, or IP-based protection sees every request as coming from the proxy.
    • Researcher-authored HTML in questionnaire JSON (instructions, title, text, left/right, sub-question text for grids and checklists) is run through a bleach-backed allowlist sanitizer at JSON load. Permitted tags cover text formatting, lists, tables, links, images, and <video>/<audio>/<source>/<track>; protocols are restricted to http, https, and mailto.
    • CSRF protection is enforced across the admin blueprint via a before_request hook backed by flask_wtf.
    • csv.writer-based serialisation.
    • Admin password comparison uses hmac.compare_digest at both the login and the database-delete confirmation.
    • route_database_download resolves the SQLite URI relative to current_app.root_path and refuses paths that escape it.

    Internal Refactoring

    • Introduced a service layer consolidating previously scattered logic: ParticipantService, SessionRecoveryService, ParticipantRoutingService, ParticipantQuestionnaireService, and AdminStatsService.
    • Results class moved to services/data_export.py; fixed an export-flags issue discovered during the move.
    • Added unit tests covering JSON tables, built-in model classes, results calculations, session management, page routing, participant management, and questionnaire handling.

    Bug Fixes

    • Fixed redirect_next_page edge cases.
    • Fixed condition being lost when external ID was loaded via URL arguments.
    • Fixed debug condition picker showing on projects with no conditions.
    • Fixed slider default size not applying to the right label.
    • Fixed page-list composition issue in the admin progress route.
    • Fallback source for standard deviation/variance now uses the population method.
  • 2024-05-09

    BOFS 2.0 released

    BOFS 2.0 brings a new BOFS CLI command, an updated admin panel with summary statistics and per-condition box plots, JSON-defined custom data tables, and compatibility with the latest Flask and SQLAlchemy. Archived on Zenodo (DOI 10.5281/zenodo.11176739).

    Core changes

    • Removed support for Python 2.7.
    • Compatible with the latest Flask and SQLAlchemy.
    • New documentation site.
    • Updated all JavaScript libraries.
    • Consent text now lives in consent.html next to the configuration file (was inline in the config).
    • Added a BOFS CLI command to start projects — no more run.py.
    • Removed the requirement for an app directory in the project.
    • flask-compress now serves brotli and gz files (e.g. for Unity WebGL builds).
    • Improved auto-reload that picks up template changes.
    • Export code rewritten to minimize the chance of throwing exceptions.
    • Friendly error messages in production mode instead of a generic 500 page.
    • 500 errors are saved to a log file when not in debug mode.
    • On startup, missing database columns from tables and questionnaires are added automatically (SQLite only).
    • Web crawlers are detected and excluded from condition counts and the progress page.
    • Many bug fixes.

    Project configuration

    • Specify a fixed completion code, or redirect users to an external URL when they reach /end.
    • /start_mturk renamed to /external_id.
    • The prompt on the /external_id page is now configurable.
    • If a PROLIFIC_PID URL parameter is present, the session variable is set and the input field auto-populates.
    • Conditions are defined with labels (rather than numbers), can be selectively disabled, and labels show on the progress page and in exports.
    • New ABANDONED_MINUTES and COUNTS_INCLUDE_ABANDONED options for ignoring inactive participants in condition counts.

    Admin panel

    • Download the SQLite database from the admin panel, or clear it.
    • Admin panel styling refresh.
    • Admin login redirects to the originally requested page instead of always going to the progress page.
    • Mark individual participants as excluded from condition counts.
    • New results page with summary statistics for any numeric data.

    Tables, questionnaires, and blueprints

    • Define custom exports inside JSON-defined tables.
    • HAVING clauses are now supported in custom exports.
    • Configure the default value of sliders.
    • Mobile rendering improvements for questionnaires.
    • The /table POST route now accepts JSON data.
    • Blueprints no longer require a views.py.
    • New "simple" HTML pages — templates that don't need any Python code.
    • Custom question types can be defined for use within questionnaires.
    • Questionnaires and tables can live inside a blueprint's folder.
